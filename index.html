📌 計費原則總覽

GCP 多數服務採即付即用模式，計費維度依產品而異，可分為四大模型：運算型（Compute）、容量型（Storage）、操作型（API/IO）、網路型（Egress）。建議將每個服務拆成「計費單位、主要成本驅動、隱藏成本、是否預配型、是否有折扣」五個欄位進行 FinOps 管理。

📋 FinOps 成本模型分類

🔴 預配型服務風險

Compute Engine、Cloud SQL、Memorystore、Filestore 等預配型服務的核心風險是閒置資源持續計費。建議定期審查未使用的實例並及時關閉或縮減規格。

🟡 請求型服務風險

Cloud Run、Firestore、Cloud KMS、Secret Manager 等請求型服務的核心風險是高流量或高頻操作導致費用快速累積。建議設定預算警示並監控請求量。

🔵 儲存型服務注意

Cloud Storage 要注意儲存類別最短保留時間、小檔案大量操作與跨區流量。Archive 類別雖然儲存便宜，但取出費用高，需依存取頻率選擇適合的儲存類別。

🟢 AI 服務特別注意

Vertex AI 各功能模組獨立計費，需特別注意模型訓練、推論、token、查詢與資料儲存是否分開計費。端點若持續部署即使無請求也會計費。

🟣 折扣策略

Compute Engine 與 GKE 可善用承諾使用折扣（CUD）與持續使用折扣（SUD）。長期穩定工作負載建議評估 1 年或 3 年承諾方案，可節省顯著費用。

🩵 Egress 是最大隱藏成本

跨區或對外網路流量（Egress）在 Compute Engine、Cloud SQL、Cloud Storage、Load Balancing 等多個服務中都會產生費用，是最常被低估的成本。架構設計時應盡量將服務部署在同一區域。

⚠️ 常見隱藏成本清單

以下是 FinOps 實務中最常被忽略的成本來源：

未刪除的磁碟

VM 停止後，Persistent Disk 仍持續計費。刪除 VM 時務必確認是否一併刪除磁碟。

備份保留

Cloud SQL、Firestore PITR 的備份保留天數越長，儲存費用越高，需依業務需求設定合理保留期。

小檔案大量操作

Cloud Storage 的 Operations 費用在小檔案高頻寫入場景下會快速累積，建議批次處理。

網路 Egress

跨區或對外流量費用橫跨多個服務，是最常被低估的成本，需在架構設計階段就納入考量。

Firestore 索引放大

索引設計不當會大幅放大讀取次數，進而推高費用。需定期審查索引設計。

GKE Pod requests 過大

Autopilot 模式依 Pod requests 計費，設定過大的 requests 即使實際用量低也會浪費成本。
